# Backend Setup

This file explains how to prepare and run the Django backend. Project conventions are in [TUTORIAL.md](TUTORIAL.md).

## Requirements

- Python 3.14
- Docker Desktop, optional but useful for running the full app

## Local Python Setup

From the repository root:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip setuptools wheel
python -m pip install -r backend\requirements.txt
```

Copy the local environment template:

```powershell
Copy-Item backend\.env.example backend\.env
```

`backend/.env` is loaded automatically by Django in local Python runs and by Docker Compose for the backend service. Real secret values belong in `backend/.env`, not in `backend/.env.example`.

Keep comments on their own lines in `.env` files. Do not write inline comments after values, because inline comments are treated as part of the value.

## Environment Files

Use these files this way:

- `backend/.env.example`: committed template with safe placeholders.
- `backend/.env`: ignored local file for real local values and secrets.

After changing `backend/.env`, restart Django so settings are reloaded.

The main sections are:

- Core Django settings: debug mode, secret key, allowed hosts.
- Frontend and browser integration: frontend URL, CORS, CSRF trusted origins.
- Email delivery: console email or real SMTP.
- Account verification and password recovery: resend cooldowns and reset limits.
- TOTP authenticator app settings: issuer name, code length, time window.
- Single sign-on providers: Google, Microsoft, and GitHub token verification.
- Auth endpoint throttles: rate limits for sensitive auth endpoints.

## Run The Backend

Run migrations and start Django:

```powershell
cd backend
python manage.py migrate
python manage.py runserver
```

Useful local URLs:

- Health check: `http://localhost:8000/api/health/`
- Swagger UI: `http://localhost:8000/api/docs/`
- OpenAPI schema: `http://localhost:8000/api/schema/`

This project uses a custom email-based user model. If your ignored local `backend\db.sqlite3` was created before the `accounts` app existed, delete that local database and run migrations again.

## Email, SMTP, And Password Recovery

Email is used for:

- email verification codes after registration
- password reset links
- password reset confirmations
- password change confirmations

### Option 1: Console Email For Local Development

This is the safest default for local work. Emails are printed in the Django terminal instead of being sent.

```env
DJANGO_EMAIL_BACKEND=django.core.mail.backends.console.EmailBackend
DJANGO_DEFAULT_FROM_EMAIL=no-reply@catsos.local
```

Use this mode when you are building or testing account flows locally and do not need real email delivery.

### Option 2: Real SMTP

Use SMTP when you want emails to arrive in a real inbox.

First set the SMTP backend:

```env
DJANGO_EMAIL_BACKEND=django.core.mail.backends.smtp.EmailBackend
```

Then fill in your provider values:

```env
DJANGO_DEFAULT_FROM_EMAIL=CatSOS <no-reply@your-domain.example>
DJANGO_EMAIL_HOST=smtp.your-provider.example
DJANGO_EMAIL_PORT=587
DJANGO_EMAIL_HOST_USER=your-smtp-username
DJANGO_EMAIL_HOST_PASSWORD=your-smtp-password-or-app-password
DJANGO_EMAIL_USE_TLS=true
DJANGO_EMAIL_USE_SSL=false
DJANGO_EMAIL_TIMEOUT=10
```

Most providers use one of these combinations:

- Port `587` with `DJANGO_EMAIL_USE_TLS=true` and `DJANGO_EMAIL_USE_SSL=false`.
- Port `465` with `DJANGO_EMAIL_USE_TLS=false` and `DJANGO_EMAIL_USE_SSL=true`.

Do not enable TLS and SSL at the same time.

For hosted email providers, the password is often not your normal account password. Use the SMTP password or app password generated by that provider.

### Resend SMTP Example

Use Resend SMTP when you have a verified sending domain and a Resend API key.
Put the real API key only in `backend/.env`.

```env
DJANGO_EMAIL_BACKEND=django.core.mail.backends.smtp.EmailBackend
DJANGO_DEFAULT_FROM_EMAIL=CatSOS <no-reply@your-domain.example>
DJANGO_EMAIL_HOST=smtp.resend.com
DJANGO_EMAIL_PORT=587
DJANGO_EMAIL_HOST_USER=resend
DJANGO_EMAIL_HOST_PASSWORD=your-resend-api-key
DJANGO_EMAIL_USE_TLS=true
DJANGO_EMAIL_USE_SSL=false
DJANGO_EMAIL_TIMEOUT=10
```

Replace `no-reply@your-domain.example` with an address on your verified domain.
Do not commit the Resend API key.

### Gmail SMTP Example

For a personal Gmail account or a Google Workspace mailbox, use Gmail's SMTP server and a Google app password.

Recommended STARTTLS setup:

```env
DJANGO_EMAIL_BACKEND=django.core.mail.backends.smtp.EmailBackend
DJANGO_DEFAULT_FROM_EMAIL=CatSOS <your.gmail.address@gmail.com>
DJANGO_EMAIL_HOST=smtp.gmail.com
DJANGO_EMAIL_PORT=587
DJANGO_EMAIL_HOST_USER=your.gmail.address@gmail.com
DJANGO_EMAIL_HOST_PASSWORD=your-16-character-google-app-password
DJANGO_EMAIL_USE_TLS=true
DJANGO_EMAIL_USE_SSL=false
DJANGO_EMAIL_TIMEOUT=10
```

SSL alternative:

```env
DJANGO_EMAIL_BACKEND=django.core.mail.backends.smtp.EmailBackend
DJANGO_DEFAULT_FROM_EMAIL=CatSOS <your.gmail.address@gmail.com>
DJANGO_EMAIL_HOST=smtp.gmail.com
DJANGO_EMAIL_PORT=465
DJANGO_EMAIL_HOST_USER=your.gmail.address@gmail.com
DJANGO_EMAIL_HOST_PASSWORD=your-16-character-google-app-password
DJANGO_EMAIL_USE_TLS=false
DJANGO_EMAIL_USE_SSL=true
DJANGO_EMAIL_TIMEOUT=10
```

For Google Workspace with a custom domain, use the full Workspace mailbox instead:

```env
DJANGO_DEFAULT_FROM_EMAIL=CatSOS <no-reply@your-domain.example>
DJANGO_EMAIL_HOST_USER=no-reply@your-domain.example
```

Gmail setup checklist:

1. Turn on 2-Step Verification for the Gmail or Google Workspace account.
2. Create a Google app password for CatSOS.
3. Paste the app password into `DJANGO_EMAIL_HOST_PASSWORD`.
4. Use the full email address as `DJANGO_EMAIL_HOST_USER`.
5. Restart Django after editing `backend/.env`.
6. Send a test email with the command in the next section.

Do not use your normal Google account password in `DJANGO_EMAIL_HOST_PASSWORD`. Google app passwords are separate 16-character passwords for apps that cannot use "Sign in with Google". Google may hide the app password option for some accounts, including some work or school accounts, accounts that only use security keys for 2-Step Verification, and accounts enrolled in Advanced Protection. In those cases, use a different SMTP provider or ask the Workspace administrator to configure an allowed sending method.

Gmail is good enough for local testing and small demos. For production-style transactional email, prefer a dedicated email provider because Gmail has sending limits and account-security controls that can interrupt automated delivery.

### Test Email Delivery

Start Django with your `.env` loaded, then run:

```powershell
cd backend
python manage.py shell -c "from django.core.mail import send_mail; send_mail('CatSOS email test', 'If you can read this, CatSOS email works.', None, ['you@example.com'])"
```

Replace `you@example.com` with your test recipient.

If you use console email, the message should appear in the terminal. If you use SMTP, the message should arrive in the recipient inbox.

### Password Reset URLs

Password reset emails use `DJANGO_FRONTEND_URL` to build links:

```env
DJANGO_FRONTEND_URL=http://localhost:5173
```

For production, set this to the real frontend origin, for example:

```env
DJANGO_FRONTEND_URL=https://catsos.example.com
```

## TOTP Authenticator App Setup

TOTP means time-based one-time password. Users scan a QR code with an authenticator app and then enter a 6-digit code when the backend asks for it.

Examples of compatible apps:

- Google Authenticator
- Microsoft Authenticator
- 1Password
- Bitwarden
- Authy

### Recommended Local Values

Keep these defaults unless you have a specific reason to change them:

```env
DJANGO_TOTP_ISSUER_NAME=CatSOS
DJANGO_TOTP_STEP_SECONDS=30
DJANGO_TOTP_DIGITS=6
DJANGO_TOTP_WINDOW=1
```

What each value means:

- `DJANGO_TOTP_ISSUER_NAME`: the app name shown inside the authenticator app.
- `DJANGO_TOTP_STEP_SECONDS`: how long each generated code is valid.
- `DJANGO_TOTP_DIGITS`: how many digits the code has.
- `DJANGO_TOTP_WINDOW`: how much clock drift the backend accepts. `1` accepts the previous, current, and next time window.

Do not casually change these after users have enrolled TOTP. Changing the period or digit count can make existing authenticator app entries fail.

### TOTP API Flow

All TOTP setup endpoints require a logged-in user with a JWT access token.

Use this header:

```http
Authorization: Bearer <access-token>
```

Start setup:

```http
POST /api/auth/totp/setup/
```

Body:

```json
{
  "current_password": "user-password"
}
```

The response contains:

- `secret`: a manual setup key
- `otpauth_url`: the value the frontend should turn into a QR code

Confirm setup:

```http
POST /api/auth/totp/confirm/
```

Body:

```json
{
  "totp_code": "123456"
}
```

Disable TOTP:

```http
POST /api/auth/totp/disable/
```

Body:

```json
{
  "current_password": "user-password",
  "totp_code": "123456"
}
```

When TOTP is enabled, these actions require `totp_code`:

- password login
- password change
- linking an SSO provider to an existing account

Password login body with TOTP:

```json
{
  "email": "owner@example.com",
  "password": "user-password",
  "totp_code": "123456"
}
```

## SSO Setup

The backend supports SSO token verification for:

- Google
- Microsoft
- GitHub

The backend does not show provider login buttons by itself. The frontend must get a provider token from the provider SDK or OAuth flow, then send that token to the backend.

SSO login or signup:

```http
POST /api/auth/sso/login/
```

Link an SSO provider to an already logged-in account:

```http
POST /api/auth/sso/link/
```

Common request body:

```json
{
  "provider": "google",
  "token": "provider-token-from-frontend"
}
```

Allowed `provider` values:

- `google`
- `microsoft`
- `github`

If the CatSOS account already exists with the same email, `/api/auth/sso/login/` rejects the request and asks the user to sign in normally and link the provider. This prevents someone from taking over an existing email/password account just by presenting a provider token with the same email.

### Google SSO

Use Google SSO when the frontend can obtain a Google ID token. This CatSOS backend verifies Google ID tokens; it does not need a Google client secret for the current login flow.

Google Cloud setup:

1. Open Google Cloud Console.
2. Create or select the project that belongs to CatSOS.
3. Open `APIs & Services` or `Google Auth Platform`.
4. Configure the branding or OAuth consent screen:
   - App name: `CatSOS`
   - User support email: your team email
   - Developer contact email: your team email
5. Configure the audience:
   - Use `External` for normal Gmail users.
   - Use `Internal` only if this is restricted to one Google Workspace organization.
   - If the app is in testing mode, add your Gmail account as a test user.
6. Open `Clients`.
7. Click `Create client`.
8. Choose application type `Web application`.
9. Name it something clear, for example `CatSOS local frontend`.
10. Add the local frontend origin under `Authorized JavaScript origins`:

```text
http://localhost:5173
```

11. If the frontend uses a redirect callback route instead of a popup/button callback, add the callback under `Authorized redirect URIs`, for example:

```text
http://localhost:5173/auth/callback/google
```

12. Save the client.
13. Copy the generated client ID. It should look like this:

```text
1234567890-abc123def456.apps.googleusercontent.com
```

14. Paste that value into `backend/.env`:

```env
GOOGLE_OAUTH_CLIENT_ID=your-google-web-client-id.apps.googleusercontent.com
```

This value must be the Google client ID, not the Google client secret. The client ID and client secret are different fields in Google Cloud; CatSOS only needs the client ID so it can verify that incoming Google ID tokens were issued for this app.

Frontend behavior:

1. Load Google Identity Services in the frontend.
2. Initialize it with the same client ID you put in `GOOGLE_OAUTH_CLIENT_ID`.
3. Ask Google to sign in the user.
4. Read the Google ID token from the credential response. In Google Identity Services this is the `credential` value.
5. Send that ID token to CatSOS:

```json
{
  "provider": "google",
  "token": "google-id-token"
}
```

Backend behavior:

- verifies the token signature using Google's Python library
- checks the token audience against `GOOGLE_OAUTH_CLIENT_ID`
- requires a verified Google email address
- creates or finds the matching CatSOS social account

### Microsoft SSO

Use Microsoft SSO when the frontend can obtain a Microsoft ID token from Microsoft identity platform.

Microsoft Entra setup:

1. Open the Microsoft Entra admin center: `https://entra.microsoft.com`.
2. Sign in with a Microsoft account that is allowed to create app registrations. If this is a school or company account and app registration is blocked, ask the tenant administrator.
3. Use one of these routes to reach app registrations:
   - Fastest route: use the search bar at the top and search for `App registrations`.
   - Current Microsoft quickstart route: open `Entra ID`, then `App registrations`.
   - Older/alternate menu route: open `Identity`, then `Applications`, then `App registrations`.
   - Azure Portal fallback: open `https://portal.azure.com`, search for `Microsoft Entra ID`, then open `App registrations`.
4. Click `New registration`.
5. Name it something clear, for example `CatSOS local frontend`.
6. Choose supported account types:
   - Use `Accounts in any organizational directory and personal Microsoft accounts` if normal public Microsoft accounts should be able to sign in.
   - Use `Accounts in this organizational directory only` only if this is limited to one school or company tenant.
7. Under `Redirect URI`, choose `Single-page application (SPA)`.
8. Add the frontend redirect URI:

```text
http://localhost:5173/auth/callback/microsoft
```

9. Click `Register`.
10. On the app overview page, copy `Application (client) ID`.
11. Paste that value into `backend/.env`:

```env
MICROSOFT_OAUTH_CLIENT_ID=your-microsoft-application-client-id
```

If you still cannot find `App registrations`, your account probably does not have permission in that tenant. Use a personal Microsoft account, switch tenant from the top-right account/directory menu, or ask the tenant admin to allow app registration or assign you the Application Developer role.

Do not create or paste a Microsoft client secret for the current CatSOS frontend login flow. The frontend should use an OAuth/OIDC browser flow such as MSAL with PKCE and send the resulting Microsoft ID token to the backend.

Keep the common JWKS URL for multi-tenant login:

```env
MICROSOFT_JWKS_URL=https://login.microsoftonline.com/common/discovery/v2.0/keys
```

For a single tenant, use a tenant-specific JWKS URL:

```env
MICROSOFT_JWKS_URL=https://login.microsoftonline.com/<tenant-id>/discovery/v2.0/keys
```

Frontend behavior:

1. Ask Microsoft to sign in the user.
2. Read the Microsoft ID token.
3. Send that ID token to CatSOS:

```json
{
  "provider": "microsoft",
  "token": "microsoft-id-token"
}
```

Backend behavior:

- reads signing keys from `MICROSOFT_JWKS_URL`
- verifies the token signature
- checks the token audience against `MICROSOFT_OAUTH_CLIENT_ID`
- checks the issuer shape
- uses `email` or `preferred_username` as the CatSOS email

### GitHub SSO

The backend can verify a GitHub access token by calling:

- `https://api.github.com/user`
- `https://api.github.com/user/emails`

It then requires the GitHub account to have a verified primary email.

Current request body:

```json
{
  "provider": "github",
  "token": "github-access-token"
}
```

Important production note: a normal GitHub OAuth web flow requires a client secret to exchange an authorization code for an access token. Do not put a GitHub client secret in the frontend. For production-quality GitHub login, add a backend callback/code-exchange endpoint and keep the GitHub client secret on the backend only.

For the current MVP, Google or Microsoft is simpler because this backend already verifies ID tokens directly.

### SSO And TOTP Together

SSO provider linking requires `totp_code` when the logged-in user has TOTP enabled:

```json
{
  "provider": "google",
  "token": "google-id-token",
  "totp_code": "123456"
}
```

Current limitation: SSO login itself does not ask for a CatSOS TOTP code. TOTP is enforced for password login, password change, and SSO provider linking. If you want CatSOS TOTP to protect SSO login too, add a second-step challenge flow before returning JWTs.

## CORS And CSRF For Local Frontend

For local React development, keep these values:

```env
DJANGO_FRONTEND_URL=http://localhost:5173
DJANGO_CORS_ALLOWED_ORIGINS=http://localhost:5173
DJANGO_CSRF_TRUSTED_ORIGINS=http://localhost:5173
```

For production, replace them with the real frontend origin. Do not use wildcard CORS in production.

## Docker Setup

Run the full app:

```powershell
docker compose up --build
```

Run only the backend service:

```powershell
docker compose up --build backend
```

Docker Compose reads `backend/.env` for the backend service. Values explicitly listed in `docker-compose.yml` override values from `backend/.env`.

Run backend CI checks in containers:

```powershell
docker compose -f docker-compose.ci.yml run --rm backend-test
docker compose -f docker-compose.ci.yml run --rm backend-openapi
```

## Common Problems

### My `.env` changes are ignored

Restart Django. Settings are read when the process starts.

Also check that you edited `backend/.env`, not a different `.env` file.

### Emails are not arriving

Check:

- `DJANGO_EMAIL_BACKEND` is `django.core.mail.backends.smtp.EmailBackend`.
- SMTP host, port, username, and password are correct.
- TLS and SSL are not both enabled.
- Your provider allows SMTP access.
- The email did not land in spam.

Switch back to console email to confirm the app is trying to send:

```env
DJANGO_EMAIL_BACKEND=django.core.mail.backends.console.EmailBackend
```

### Google SSO says the token is invalid

Check:

- The frontend is sending a Google ID token, not an access token.
- `GOOGLE_OAUTH_CLIENT_ID` is the web client ID used by the frontend.
- The Google OAuth consent screen and authorized origins include the frontend URL.

### Microsoft SSO says the token is invalid

Check:

- The frontend is sending a Microsoft ID token, not a Graph access token.
- `MICROSOFT_OAUTH_CLIENT_ID` matches the app registration client ID.
- `MICROSOFT_JWKS_URL` matches the tenant mode you want.

### TOTP codes are rejected

Check:

- The user's device clock is correct.
- The backend server clock is correct.
- The user scanned the latest setup QR code.
- `DJANGO_TOTP_STEP_SECONDS` and `DJANGO_TOTP_DIGITS` were not changed after setup.

If local testing is blocked, disable TOTP for the user in Django Admin or the database and enroll again.
